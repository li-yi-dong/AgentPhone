## 设计审查报告 #001

**主题**: AgentPhone 初始架构设计（v0.3.0）— PTY 透明代理 + 局域网直连双层架构
**日期**: 2026-07-22
**版本**: DESIGN.md v0.3.0
**结论**: ⚠️ 有条件批准

---

### 核心约束检查

- [x] **本地终端体验零感知**：架构意图正确——Desktop Agent 作为 PTY Master 透明包裹 Claude Code，本地终端感知不到中间层的存在。但 PTY Resize 冲突（见高风险 #1）在当前设计下会直接破坏此约束，**批准条件：必须先解决 Resize 冲突**。
- [x] **PTY 代理透明性（不修改 I/O）**：设计明确为"透明包裹"，输出广播直接转发原始 ANSI 字节，输入合并串行写入，未做内容修改。满足。
- [ ] **延迟 < 50ms（本地路径）**：**⚠️ 未验证。** 设计未提供延迟分析。本地路径新增了一个 Node.js 进程作为 I/O 中间层，需要实测数据（尤其在大量输出场景下的背压行为）。
- [x] **Broker 侧无 PTY 内容持久化**：MVP 阶段无 Broker，不适用。局域网直连设计下内容仅在 Desktop Agent 内存缓冲，满足精神。

---

### 风险与问题

#### 高风险

**1. PTY Resize 冲突——直接违反"本地终端零感知"核心约束**

当前设计中 `PTY_RESIZE` 消息（手机端 → Desktop Agent）直接调整 PTY 的 cols/rows。这产生以下问题：

- 手机竖屏时 cols 通常为 40–80，本地终端可能为 220 cols。手机发送 resize 后，PTY 的实际尺寸变小，Claude Code 会重新排版输出（折行、清屏），本地终端会出现显示错乱。
- 本地终端物理窗口没变，但 PTY 告知 Claude Code 的尺寸与本地终端不符，违反"零感知"约束。
- 本地终端自身 resize（SIGWINCH）也会发送给 PTY，与手机端 resize 产生竞争。

**影响**：P0 核心约束被破坏，Claude Code 在本地终端的输出排版失真。

**建议**：PTY 的实际尺寸必须以本地终端为唯一权威来源。手机端 `PTY_RESIZE` 不应改变 PTY 实际大小，只应传递给手机端 xterm.js 作为本地渲染参数（调整 xterm.js 视口 cols/rows，由 xterm.js 自行处理滚动和折行）。具体协议修改建议：将消息类型拆分为 `PTY_RESIZE`（本地终端专用，Desktop Agent 监听 SIGWINCH 自动处理）和 `MOBILE_VIEWPORT`（手机端视口声明，仅影响客户端渲染，不传入 PTY）。

---

**2. WebSocket 明文传输导致 Token 可被局域网嗅探**

MVP 使用 `ws://`（明文），二维码中包含 token，AUTH_REQUEST 消息携带 token 明文传输。在公司网络、咖啡厅 WiFi 等共享局域网中，任何具有混杂模式抓包能力的设备均可截获 token 并伪造连接。

**影响**：攻击者可获取 Claude Code 完整终端控制权，包括执行任意命令。

**建议**：
- 短期（MVP）：token 应足够长（建议 ≥ 128 bit 随机），并在连接握手时使用 token 作为 HMAC 密钥对 challenge 签名，而非直接传输明文 token。或最低限度：token 通过二维码仅展示一次，不在网络中明文传输（改为用 token 做握手 challenge-response）。
- 中期：升级为 `wss://`（自签名证书 + 证书指纹固定，在二维码中包含指纹）。

---

#### 中风险

**3. Token "一次性"语义歧义，实际可被重放**

DESIGN.md 5.4 节描述 token 为"一次性使用"，但同时说明"手机断线重连需重新扫码或使用缓存 token"。"缓存 token 可重用"与"一次性"矛盾。

**影响**：若 token 被截获，攻击者可在 token 有效期内（Desktop Agent 运行期间）随时重放连接。无法做到真正的一次性防重放。

**建议**：明确 token 的生命周期语义：Desktop Agent 重启失效（已满足）+ 同一 token 同时只允许一个活跃连接（已有意图）。如需断线重连，允许相同 token 重新建立连接是合理的，但应在文档中移除"一次性"措辞，改为"会话级 token，仅在 Desktop Agent 运行期间有效，每次启动重新生成"。

---

**4. 历史缓冲回放的 ANSI 序列完整性**

`STATE_SNAPSHOT` 中 `history` 字段为原始 ANSI 字节流的拼接字符串。原始终端输出流在任意位置截断可能导致 ANSI 转义序列不完整（例如在 `\x1b[32m` 中途截断），手机端 xterm.js 渲染时会出现颜色/光标状态异常，且该状态会"污染"后续所有输出的渲染。

**影响**：手机中途加入时终端显示异常，用户体验降级。

**建议**：历史缓冲不应存储原始字节流快照，而应使用 xterm.js 的 `serializeAddon`（Terminal State Serialization）导出已渲染的终端 buffer 状态，或在 Desktop Agent 侧维护一个 xterm.js 实例（headless）来持续消费输出并序列化状态。另一个可行的 MVP 方案是：回放时重置 xterm.js 状态（`\x1bc` 全清）再发送历史，避免不完整序列问题。

---

**5. 多手机连接时 PTY_RESIZE 竞争未定义**

设计支持多手机同时连接（架构图显示"所有已连接的 WebSocket 客户端"），但协议中 `PTY_RESIZE` 来自任意手机端，多部手机不同屏幕方向同时发送 resize 会导致 PTY 尺寸反复抖动。

**影响**：多用户场景下终端排版持续抖动，影响所有端（本地终端和所有手机）。

**建议**：结合高风险 #1 的方案，手机端 resize 不应影响 PTY 实际尺寸。此问题在修复高风险 #1 后自动消除。如未修复，应在 MVP 阶段限制只接受最后连接的手机端的 resize 消息，并在文档中注明此限制。

---

**6. 输入来源无标识，无法实现差异化处理和审计**

当前协议中，本地终端输入和手机端输入串行写入 PTY，无来源标识。这在当前 MVP 范围内是可接受的，但会阻碍后续功能（如手机端"只读观察模式"、输入审计日志、优先级控制）。

**影响**：短期功能受限，Phase 2 的"审批流程"（P1）实现时需要区分输入来源。

**建议**：Desktop Agent 内部维护输入来源枚举（`LOCAL` / `MOBILE:{clientId}`），写入 PTY 时附加来源元数据（仅内部使用，不修改 PTY I/O 内容本身）。协议层暂不需要变更，内部实现提前预留。

---

#### 低风险 / 建议优化

**7. 本地路径延迟缺乏量化基准**

设计给出了 50ms 的延迟约束，但没有分析 Desktop Agent 作为 PTY 中间层引入的额外延迟。在大量输出场景（如 `cat` 大文件）下，广播给 WebSocket 客户端的操作如果产生背压，是否会通过事件循环间接延迟本地终端写入，需要验证。

**建议**：Phase 1 完成后，补充本地路径延迟基准测试（测量 PTY 读取到本地终端 write 的时间），确认在大量输出（>1MB/s）下本地路径延迟可控。同时确认 node-pty 的输出读取和本地终端写入不会被 WebSocket 背压阻塞（应使用非阻塞写入 + 手机端消息队列）。

---

**8. macOS 防火墙弹窗影响首次启动体验**

Desktop Agent 监听 `:8080` 端口时，macOS 防火墙可能弹出"允许/拒绝"确认弹窗（尤其是对未签名的二进制）。用户如果拒绝或弹窗出现时未注意，WebSocket Server 无法从局域网访问。

**建议**：文档中提前说明此行为；考虑在打包时对二进制进行代码签名（`codesign`），减少防火墙拦截概率。

---

**9. 多网络接口时 IP 地址显示不明确**

Desktop Agent 启动时需要在终端打印局域网 IP 用于二维码生成。当 Mac 同时连接 WiFi 和有线以太网时，存在多个局域网 IP，需要确定选取规则（优先 WiFi 接口，或列出所有接口让用户选择）。

**建议**：实现时明确 IP 选取优先级（WiFi `en0` > Ethernet `en1` > 其他），并在存在多接口时在终端提示用户确认。

---

**10. token 长度和复杂度未规范**

DESIGN.md 中 token 仅以示例值 `xyz123` 说明，未规定实际的随机性要求。

**建议**：明确规定 token 为 ≥ 128 bit 的 CSPRNG 随机字节，以 Base62 或 Base64url 编码，总长度 ≥ 22 字符。

### 技术选型评估

| 技术 | 评估 | 结论 |
|------|------|------|
| node-pty | macOS/Linux 原生绑定，社区活跃，Claude Code 本身亦使用 Node 生态 | ✅ 合理 |
| ws 库 | 轻量无依赖，Node.js WebSocket 实际标准 | ✅ 合理 |
| xterm.js (WebView) | 完整 ANSI 支持，可行。需注意 React Native WebView 与 xterm.js 的性能边界，尤其在大量输出时 WebView 的消息传递（postMessage）可能成为瓶颈 | ⚠️ 需压测 |
| React Native + Expo | 跨平台合理选择，生态成熟 | ✅ 合理 |
| esbuild + pkg 打包 | pkg 已停止维护（archived），建议改用 `@vercel/ncc` + Node.js SEA（Single Executable Application，Node.js 21+ 原生支持）或 `bun build --compile` | ⚠️ 建议替换 |

---

### 决策

**结论：⚠️ 有条件批准**

整体架构方向正确：PTY 透明代理 + WebSocket 广播的设计思路可以满足"本地终端零感知"的核心目标，局域网直连方案技术上完全可行。

**开始实现前必须解决的前提条件（P0）：**

1. **PTY Resize 冲突（高风险 #1）**：在协议层明确区分 `PTY_RESIZE`（本地终端驱动，不暴露给手机端）和 `MOBILE_VIEWPORT`（手机端视口声明，不影响 PTY 实际尺寸）。这是启动实现前的设计修改，影响 `packages/protocol/src/messages.ts` 的消息类型定义。
2. **Token 传输安全（高风险 #2）**：在 AUTH_REQUEST 消息中改用 challenge-response 机制而非明文 token 传输，或在设计文档中明确接受此风险并记录缓解措施（如 token 足够随机、局域网场景的威胁模型说明）。

**建议在 Phase 1 期间同步解决（中风险）：**

- 修正 token"一次性"措辞（中风险 #3）
- 历史缓冲回放方案补充 ANSI 安全性说明（中风险 #4）

**可推迟到 Phase 2 处理（低风险）：**

- 延迟基准测试（低风险 #7）
- 打包工具替换（低风险，esbuild + pkg → SEA 方案）
- 输入来源内部标识预留（低风险 #6）
